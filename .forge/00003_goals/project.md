# Project Spec: Goals

## Goal

Replace the binary collection/conversation mode distinction with a unified **goals** model. A goal is a named, file-defined configuration that specifies a desired outcome, the council members to use (with their hierarchy), and the corpus to reference. Goals are processed offline (like embeddings) and selected explicitly at runtime via `--goal <name>`. The consolidated/sequential deliberation modes remain an orthogonal execution option. When this project is done, any interaction is expressed as a goal, and the core system has no hardcoded notion of "collection mode" or "conversation mode."

## Why This Matters

The current collection/conversation split is a false dichotomy — both are conversations with different intentions. Hardcoding two interaction modes limits extensibility: adding a third mode requires code changes. The goals model removes that ceiling. New interaction types are added by authoring a markdown file and running a process step, not by modifying the codebase. It also enables deployments to define arbitrarily many distinct interaction patterns, each with its own council composition and corpus scope, without touching source code.

## Architecture Overview

A `goals/` directory (configurable path) holds one markdown file per goal. Each goal file declares: a `desired_outcome` field, an ordered list of council persona references with authority tiers, and a corpus path or scope. The `goals process` command reads all goal files, validates that referenced persona files exist, and writes `goals_manifest.json`. At runtime, `--goal <name>` loads the named entry from the manifest, replacing the current logic that selects council members and corpus based on hardcoded mode. Consolidated/sequential deliberation continues to be a separate flag applied after goal loading.

## Constraints

- Goal files are markdown — no YAML-only or JSON-only goal definitions
- No goal logic (council selection, corpus scoping) hardcoded in Python source — all in goal files
- The `goals process` step must be idempotent: running it twice produces the same manifest
- Existing `--mode consolidated|sequential` flag must continue to work unchanged after this refactor
- No new external dependencies introduced
- `intake.md` and `create-plan.md` are deployment-specific examples — their content is configuration, not platform logic

## Performance Requirements

None beyond existing baselines — goal loading from manifest is a file read, not a hot path.

## Security Considerations

Goal files reference persona files by path — validate that resolved paths stay within the configured personas directory to prevent traversal.

## Out of Scope

- Automatic/inferred goal selection at runtime (goal must always be named explicitly)
- Dynamic goal instantiation at runtime (goals are always pre-processed, offline)
- UI or frontend changes
- COM-B plan logic in Python source — the `create-plan` goal drives this entirely through its goal file and council configuration
